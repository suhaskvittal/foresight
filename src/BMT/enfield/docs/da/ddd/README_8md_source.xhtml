20 ## Testing
21 
22 Enfield uses the [Google test framework](https://github.com/google/googletest) to test its components.
23 To enable automated tests, you should issue the ```cmake``` command as follows:
24 
25 ```
26 $ cmake ../ -DENABLE_TESTS=on
27 $ make && make test
28 ```
29 
30 It is possible to specify the root folder of the GTest framework. In order to do that, you should
31 pass to ```cmake``` the option ```-DGTEST_ROOT=<path-to-gtest-libs>```.

33 ## Hacking
34 
35 Even though this project is pretty new, it was designed to be extensible. So, here are a few tips
36 in order to implement your own algorithm to your desired architecture. Below, I'll list some classes
37 that are important to be aware of.
38 
39 (Note that this is a 'begginers guide', so you can do more stuff once you learn the code)
40 
41 * ```efd::QModule```: The core class of enfield. It holds the AST preprocessed to be easier to use, as well as some other useful methods for modifying the AST. i.e.: ```insertSwapAfter```; ```insertNodeAfter```; ```replaceAllRegsWith```, etc.
42 
43 * ```efd::QbitAllocator```: This is the base class for implementing allocators. In order to implement your own, you should extend this class and implement the method ```solveDependencies```. This method is responsible for inserting swaps based on the dependencies (that are given by parameter).
44 
45 * ```IBMQX2.def```: This is not a class per se. Take a look in this file in order to create the specification of other architectures. You should use macros like ```EFD_ARCHITECTURE```, and inserting the line: ```#include "enfield/Arch/YourArch.def"``` in the files ```enfield/Arch/Architecture.h``` and ```lib/Arch/Architecture.cpp``` right bellow the ```IBMQX2.def```.
